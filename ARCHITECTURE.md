# Architecture

The Bu1ld is a TanStack Start (Vite + React) application deployed as a Cloudflare Worker with a Supabase Postgres backend.

## Surfaces

| Surface                 | Audience                              | Entry                                                                                                                 |
| ----------------------- | ------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| Public institution site | Visitors                              | `/`, `/labs`, `/programs-public`, `/competitions`, `/people`, `/partnerships`, `/publications`, `/apply`, `/evidence` |
| Auth                    | Applicants                            | `/signup`, `/login`, `/auth/callback`                                                                                 |
| Member portal           | Authenticated members                 | `/dashboard`, projects, research, programs, invitations                                                               |
| Admin console           | Admins / institutional administrators | `/admin`                                                                                                              |

## Stack

- **UI:** React 19, TanStack Router / Start, Tailwind 4, Framer Motion
- **Data:** Supabase Auth + Postgres + RLS
- **Deploy:** Vite build → `dist/server/server.js` + `dist/client` assets → Wrangler (`bu1ld-landing`)
- **Email:** `/api/email` (Resend) with session or shared-secret auth

## Auth model

1. **Legacy `profiles.role`:** `member` | `project_lead` | `admin` — used by most RLS policies.
2. **Institutional `member_roles`:** additive grants (`researcher`, `reviewer`, `mentor`, `lab_lead`, `startup_founder`, `administrator`, …).
3. **UI helpers:** `src/lib/roles.ts` — `isAdministrator` accepts legacy admin **or** institutional administrator.
4. **Authorization truth:** Supabase RLS + security-definer RPCs (e.g. `review_project_application`, `accept_invitation`) and `is_platform_admin()`. Frontend gates are UX only.

## Data flow (happy path)

1. Visitor discovers labs/programs on public pages → `/apply` → `/signup`.
2. Onboarding completes profile → dashboard.
3. Apply to project or program → lead/admin reviews via RPC → `project_memberships` / acceptance.
4. Team invites via `invitations` → `/invitations` accept/decline.
5. Members submit contributions / milestones → lead verification.
6. Admins moderate claims, members, content in `/admin`.

## Runtime configuration

- Build-time: `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` (Vite inlines).
- Fallback: `/runtime-env.js` generated by `scripts/generate-runtime-env.mjs` from `.env` or the deployment environment. Public keys are not committed in `wrangler.jsonc`.
- Server secrets: service role, Resend, digest secret — never `VITE_`-prefixed.
